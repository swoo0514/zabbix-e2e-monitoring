## #1. 분리 컨테이너 환경에서 Zabbix Agent 접속 불가 (Passive Check 실패)

- **발생일:** 2026-06-30 (Day 1, 스택 최초 기동 직후)
- **증상:** Web UI의 Problems에 `Zabbix server / Linux: Zabbix agent is not available (for 3m)` 경고가 상시 표시됨.
- **환경:** 단일 VM에서 Docker Compose로 `zabbix-server`와 `zabbix-agent2`를 독립 컨테이너로 분리 운영.

### 원인 분석
- Zabbix Server 호스트 설정의 Agent 인터페이스 주소가 루프백 주소 `127.0.0.1`로 설정되어 있었음.
- 단일 호스트에 agent가 함께 설치된 전통적 구성과 달리, **컨테이너 분리 구성에서는 Server 컨테이너 내부의 `127.0.0.1`이 Server 컨테이너 자신**을 가리키므로 격리된 Agent 컨테이너에 도달할 수 없음.
- 통신 방향별로 분리하여 진단:
  - **Active Check** (Agent → Server:10051): 정상. Agent 로그에서 `active check ... is working again` 확인.
  - **Passive Check** (Server → Agent:10050, 호스트 인터페이스 주소 사용): 도달 불가로 가용성 검증 실패, 경고 발생.

### 해결 방안
- `Data collection → Hosts → "Zabbix server" → Interfaces → Agent` 인터페이스 설정 수정:
  - Connect to: **DNS**
  - DNS: **`zabbix-agent2`** (Docker Compose 서비스명 = 내장 네트워크의 DNS 별칭)
  - Port: `10050` 유지
- 설정 반영 후 폴링 주기(1~3분) 내에 ZBX 가용성 아이콘이 정상(녹색)으로 전환되고 관련 경고가 RESOLVED됨.

### 배운 점
- 컨테이너 환경에서는 재시작 시 내부 IP가 변경되므로, 호스트 인터페이스 주소는 고정 IP가 아닌 **서비스명(DNS)으로 매핑**해야 구성이 컨테이너 재생성 후에도 유효하게 유지된다.
- Agent 가용성 이슈는 **Active Check와 Passive Check의 통신 방향을 분리하여 단계적으로 진단**한다.

## #2. Nginx Basic Auth 미작동 (Rewrite 단계와 Access 단계의 실행 순서)

- **발생일:** 2026-07-01 (nginx `/secure` 경로 고도화 검증 중)
- **증상:** `/secure`에 인증 정보 없이 요청했으나 `401 Unauthorized`가 아닌 `200 OK ("SECURE OK")`가 반환되어 접근 제어가 우회됨.
- **환경:** `location = /secure` 블록에 `auth_basic` 지시어와 `return 200 "SECURE OK";` 지시어를 혼용 적용.

### 원인 분석
- Nginx의 HTTP 요청 처리는 **Rewrite → Access → Content** 단계 순으로 실행됨.
- `return` 지시어(ngx_http_rewrite_module)는 선행 단계인 **Rewrite 단계에서 요청 처리를 즉시 종료**하고 응답을 전송함.
- `auth_basic` 인증 검사는 이후의 **Access 단계**에서 실행되므로, `return`의 조기 종료에 의해 인증 루틴 자체가 생략됨.

### 해결 방안
- 해당 블록에서 `return` 지시어를 제거하고 실제 파일 서빙(`alias`) 방식으로 변경 — Access 단계의 인증을 통과한 후 Content 단계에서 정적 파일이 서빙되도록 함.
- 보호 대상 파일(`secure.txt`)은 웹 루트 외부 경로(`/etc/nginx/auth`)에 배치하여 인증 경로 외의 직접 접근 가능성을 배제함.

### 배운 점
- Nginx 설정은 구성 파일상의 표기 순서가 아니라 **내부 처리 단계(Phase)의 우선순위**를 기준으로 해석해야 하며, 인증이 필요한 블록에서는 `return` 지시어 사용을 지양한다.

## #3. 내장 API 명세 오인에 따른 수동 구현 및 스크립트 전면 리팩토링

- **발생일:** 2026-07-02 (midibus Browser Item 구현 중)
- **증상:** `findElement`가 요소 등장 전 `null`을 반환하는 것을 보고 요소 대기용 수동 `for` 폴링 루프를 구현하고, 네이티브 `confirm` 창도 우회 스크립트를 자작함. 코드 복잡도가 증가하고 실행 안정성이 저하됨.

### 원인 분석
- Browser Item API 규격을 요약 자료 위주로 파악하여, 내장 기능인 `setElementWaitTimeout`(암묵적 대기), `getAlert()`, `collectPerfEntries()` 등의 메서드가 부재한 것으로 오인함.
- 해당 메서드 전체 목록은 메인 가이드가 아닌 별도 전용 문서(`preprocessing/javascript/browser_item_javascript_objects`)에 분리 수록되어 있어 확인이 누락됨.

### 해결 방안
- 공식 레퍼런스 원문을 확인한 뒤 내장 API 기반으로 전면 리팩토링:
  - 수동 폴링 루프 → `setElementWaitTimeout`(암묵적 대기)으로 대체
  - confirm 우회 → `unhandledPromptBehavior` 옵션 및 `getAlert().accept()` 적용
  - 스텝별 성능 계측 → `collectPerfEntries("라벨")` 기반 표준 마킹 적용
- 리팩토링 후 시나리오 총 소요 시간이 230초에서 약 26초로 감소, 이후 용량 해석의 기준값이 됨.

### 배운 점
- 프레임워크 기능의 부재를 단정하고 커스텀 로직을 자작하기 전에 **공식 레퍼런스 원문을 검증**해야 하며, 표준 내장 API의 활용이 성능과 유지보수성에 직결된다.

## #4. Web UI 에디터 입력 제약 우회를 위한 API 배포 프로세스 전환

- **발생일:** 2026-07-02
- **증상:** Web UI의 Browser Item 스크립트 편집기에 긴 JavaScript 소스를 붙여넣을 때 일부 구문이 유실·손상되며 `parse error at line N` 오류가 지속 발생함.

### 원인 분석
- 웹 에디터 경유 시 재현되는 손상으로, 정확한 손상 지점은 특정하지 않음(에디터를 경유하지 않으면 미발생).
- 별도로, 주석 내 유니코드 기호(예: `→`)가 Duktape 스크립트 엔진의 구문 분석 오류를 유발하는 사례를 확정함.

### 해결 방안
- 웹 편집기를 경유하지 않고 Zabbix API(`item.update`)로 로컬 파일의 스크립트를 직접 배포하는 셸 스크립트(`zabbix/update-item-script.sh`)를 구축함(`jq -Rs`로 안전한 JSON 문자열 가공 후 전송).
- 소스 내 주석을 포함한 문자열을 ASCII 규격으로 통일하여 파서 호환성 확보.

### 배운 점
- GUI 관리 도구의 입력 제약은 **API 기반 배포 자동화**로 우회할 수 있으며, 부수 효과로 설정을 코드로 버전 관리하는 체계(Config-as-Code)를 확보하게 된다.

## #5. 보안키 재생 검증의 허용 IP 설정 기준 확립 (실행 주체의 Egress IP)

- **발생일:** 2026-07-03 (Step 4 보안키 구현 중)
- **증상:** 보안키에 허용 IP를 설정하면 해당 키가 적용된 재생 URL이 그 IP에서만 재생됨. 자동화 검증 시 어떤 IP를 허용값으로 등록해야 하는지가 문제로 대두됨.

### 원인 분석
- E2E 시나리오에서 재생 URL을 호출하는 실제 주체는 작업자의 로컬 브라우저가 아니라 **인프라 내부의 Selenium 컨테이너**임.
- 따라서 작업자 환경에서 확인한 공인 IP를 등록하면, 실제 요청의 출발 IP인 **VM의 공인 아웃바운드 IP(Egress IP)** 와 불일치하여 재생이 차단됨.

### 해결 방안
- 허용 IP를 VM 공인 Egress IP 매크로(`{$VM.EGRESS_IP}`)로 설정 → Selenium의 출발 IP와 일치 → 재생 상태(`.jw-state-playing`) 검증 통과.

### 배운 점
- 네트워크 접근 제어가 결합된 기능을 자동화로 검증할 때는 **자동화 런타임의 아웃바운드 Egress IP**를 기준으로 허용 정책을 매핑해야 오탐을 방지할 수 있다.

## #6. Operational Data의 표현식 매크로 미해석 — 위치(Location)별 지원 명세

- **발생일:** 2026-07-07 (알림 템플릿 고도화 중)
- **증상:** 실패 지점(`err_at`)·에러 분류(`err_class`)를 알림에 싣기 위해 트리거 Operational data 필드에 표현식 매크로(`{?last(...)}`)를 삽입했으나, 값으로 해석되지 않고 문자열 리터럴 그대로 알림에 출력됨.

### 원인 분석
- Zabbix 매크로는 전역 문법이 아니라 **배치 위치(Location)별로 지원 여부가 규정**되어 있음.
- 표현식 매크로(`{?...}`)는 Event name 등 일부 위치에서만 해석되며, Operational data에서는 미지원 — 미지원 위치의 매크로는 오류 없이 리터럴로 통과됨.

### 해결 방안
- 해당 매크로를 **Event name 필드**로 이관하여 정상 해석 확인(`midibus 로그인 실패: midibus [selector: login success marker]` 형태로 알림 제목에 실패 지점·유형 노출).
- Operational data는 지원 범위 내 매크로(`{ITEM.LASTVALUE1}`)로 단순화.
- (검토 후 기각한 대안) error 아이템을 트리거 평가식에 추가해 `{ITEM.LASTVALUE2}`로 참조하는 방식 — error 아이템은 실패 시에만 값이 생기므로 이력이 없는 상태에서 핵심 트리거의 판정이 Unknown으로 전환되는 리스크가 식별되어 기각함.

### 배운 점
- 매크로가 리터럴로 출력되는 현상은 구문 오류가 아니라 **미지원 위치 배치의 신호**이며, 사용 전 공식 지원 명세표("Macros supported by location")를 기준으로 교차 검증해야 한다.

## #7. 경보 이력 데이터와 실시간 상태 지표의 시차 분석 및 라벨 명확화

- **발생일:** 2026-07-07 (알림 파이프라인 검증 중)
- **증상:** Slack 알림에는 `step=0`(실패)으로 표기되었으나, 알림의 링크로 진입한 Web UI 이벤트 상세에는 동일 필드가 `step=1`(정상)로 표시되어 동기화 오류로 해석될 여지가 발생함.

### 원인 분석
- 스크립트 결함이 아니라 **매크로 해석 시점**의 차이임:
  - 알림 메시지의 값: **장애 이벤트 발생·발송 시점의 스냅샷**을 동결하여 전송(장애 당시 기록).
  - Web UI의 값: 운영자가 **열람하는 시점의 현재값**으로 재해석(현재 상태).
- 알림 발송과 화면 열람 사이에 시나리오가 자체 복구되어 값이 0 → 1로 바뀐 정상 거동임.

### 해결 방안
- 값 보정이나 스크립트 수정은 부적절함(0=실패/1=성공 시맨틱과 트리거 식이 훼손됨).
- 혼동의 실제 원인인 라벨 모호성을 수정 — `step=0`은 "step 번호 0"으로 오독될 수 있으므로(`failed_step` 번호 체계와 충돌), 명사형 상태 표기(`login_ok={ITEM.LASTVALUE1}`)로 트리거 6종의 출력 구조를 일괄 변경함.

### 배운 점
- 외부 전송 경보는 장애 당시의 **이력(History)**, 관리 화면은 **현재 상태(Status)** 를 나타내므로, 두 값이 다르면 먼저 그 사이의 상태 변화를 확인한다.
- 지표 노출 시 값 자체보다 **라벨의 오독 가능성**을 먼저 점검한다(`step=` → `login_ok=`).

## #8. Maintenance(점검창) 적용 시 서비스/SLA 산출 제외 거동 규명 — 소스 코드 추적

- **발생일:** 2026-07-07 (Maintenance 검증 중)
- **증상:** 점검창 적용 상태에서 장애 주입(Fault Injection)을 수행한 결과, Problems에는 suppressed(눈 아이콘) 문제로 정상 표시되나 SLA Downtime 수치에는 반영되지 않는 거동 확인. "점검창은 알림만 끄고 SLA는 별도 집계된다"는 사전 예상과 상반됨.

### 원인 분석
- 층을 분리해 판별한 결과, SLA가 아니라 **서비스 층이 suppressed 문제를 무시**함(억제 중 해당 서비스가 OK 유지).
- 2026-07 기준 7.0 공식 문서 8곳 이상에서 이 규칙의 명시 조항을 확인하지 못해, Zabbix 소스 코드로 직접 추적함.
- `src/zabbix_server/service/service_manager.c`(release/7.0)의 서비스 상태 갱신 루틴 `db_update_services()` 내 심각도 집계 루프에서 다음 조건을 식별:

```c
if (NULL != event->maintenanceids)
    continue;
```

- 즉 Maintenance로 억제된 이벤트는 심각도 비교 로직 도달 전에 건너뛰어짐을 소스 레벨에서 확정함. 부가 확인: 억제 해제 시 서비스가 재계산되므로, 점검창이 닫힌 시점부터 잔존 문제의 SLA 차감이 시작됨(점검창을 열어둔 채 방치하지 않아야 하는 근거).

### 해결 방안
- 사전 계획 작업은 Maintenance 단일 구성으로 알림 억제와 SLA 보존이 함께 처리됨을 확인.
- 내장 `Excluded downtimes`는 사전 구성이 불가했던 구간의 **사후 소급 보정용**으로 역할을 재정의함.

### 배운 점
- 공식 문서에서 확인되지 않는 동작의 최종 판단 근거는 **소스 코드 분석**이다. 단, 부재 판단에는 확인 시점과 범위를 함께 명시한다.
- 가용성 평가 계층(문제 → 서비스 → SLA)의 연동 단절 지점은 층별 판별 관찰로 좁혀 추적할 수 있다.

## #9. 서버 강제 정지 실험에서의 수집 주기·수집 주체별 관측 차이 분석

- **발생일:** 2026-07-13 (Proxy 고도화 E1 — 서버 의도 정지 실험)
- **증상:** 서버를 5분 56초간 강제 정지(`docker stop zbx-server`)했으나, midibus 마스터 아이템 그래프에는 공백이 없고, 프록시로 이관한 nginx 시나리오 그래프도 서버 재기동 후 공백 없이 연속으로 표시되어 정전 검증의 판정에 혼선이 발생함.

### 원인 분석
- **수집 주기에 따른 관측 사각지대(midibus):** 마스터 아이템 주기가 10분으로, 정지 구간(09:48~09:54)이 정기 실행(09:46, 09:56) 사이에 완전히 포함되어 누락된 실행 자체가 없음. 장애 지속 시간이 수집 주기보다 짧으면 해당 아이템 관점에서는 장애가 관측되지 않는다.
- **프록시의 버퍼링·Backfill 거동(nginx):** 수집 주체가 프록시로 이관된 상태라 서버 정지 중에도 프록시가 시나리오를 계속 실행·버퍼링했고, 서버 복구 시 원본 타임스탬프로 사후 적재(Backfill)하여 그래프가 연속으로 채워짐.
- 정지 구간을 실제 공백으로 보여준 것은 **서버 자신이 수집 주체**인 10초 주기 내부 메트릭(`zabbix[wcache,history,pused]`)이었음.

### 해결 방안
- 검증 대조군을 "서버가 직접 수집하는 단주기 메트릭"으로 교체하여 정지 구간의 공백을 확정하고, 프록시 경유 메트릭의 연속성은 그래프가 아니라 **적재 데이터의 타임스탬프 목록**(1초 간격 300건 연속)으로 별도 검증함.

### 배운 점
- 동일한 장애도 **수집 주기(사각지대) × 수집 주체(Server/Proxy) × Backfill 지원 여부**의 조합에 따라 다르게 관측된다. "그래프에 공백이 없다"는 "장애가 없었다"와 동치가 아니다.
- 연속성 증빙은 시각화 그래프가 아닌 **실 데이터 타임스탬프**를 기준으로 판정한다. nodata 트리거의 여유 시간을 수집 주기의 배수로 설계해야 하는 실측 근거이기도 하다.

## #10. 임계 부하에서의 Backpressure 거동 및 고정 구간 집계 기반 측정 검증

- **발생일:** 2026-07-13 (Proxy 고도화 E3 — 128K 축소 캐시 2000 msg/s 재검증)
- **증상:** 120,000건 버스트 부하의 저장 총량을 계측하는 과정에서, 측정 시점에 따라 결과 수치가 93,960 → 98,000 → 26,000으로 변동하여 데이터 유실로 오인할 소지가 발생함.

### 원인 분석
- 히스토리 캐시 포화 조건(128K·syncer 1)에서 서버는 값을 폐기하지 않고 **수용 속도를 제어하는 Backpressure 거동**을 보였고, 이로 인해 60초분 송신의 실제 완료까지 3분 이상 소요됨(동기식 sender가 응답 대기로 감속됨을 실측). 이후 소스 추적으로 확정 — `hc_add_item_values()`(`cachehistory.c`, release/7.0)는 캐시 만석 시 "History cache is full. Sleeping for 1 second." 대기 루프로 구현되어 있어 드롭 경로가 없음.
- 검증 스크립트(`count-history.sh`)는 "현재 시점 기준 과거 N초"의 **가변 시간창(Sliding Window)** 방식이라, 도착 분포가 시간축으로 늘어진 상태에서는 시간창과 도착 구간의 상대 위치에 따라 집계값이 요동함 — 데이터가 아니라 측정 방식이 만든 착시임.

### 해결 방안
- 총량 판정을 가변 시간창 방식에서 **고정 구간 타임스탬프 버킷 집계**(DB `group by` 연산)로 전환하여 실행별 총량(60,000 / 120,000 / 120,000)과 유실 0을 확정함. 완주 판정은 시퀀스 최댓값 도달 여부로 분리함.
- 프록시 경로 측정은 설계 단계부터 2단계 카운팅(직후 + 4분 후)으로 구성 — 1차 도달 37.6% → 2차 전량 도달의 대비가 "유실이 아니라 지연 전달"의 정량 증거가 됨.

### 배운 점
- 데이터 도착이 시간축으로 늘어지는 고부하 조건에서는 시간창 기반 계측이 유실처럼 보이는 착시를 만든다. **총량 판정은 고정 구간 버킷 집계로, 완주 판정은 시퀀스 최댓값으로 분리 설계**한다.
- "캐시 포화 = 데이터 유실"이라는 통설은 실측으로 기각되었으며, 동기식 단일 sender 조건에서의 실제 거동은 드롭이 아닌 **Backpressure에 의한 송신 감속**이다(비동기·병렬 다중 소스 조건은 미검증으로 한정).
